# Roadmap: Take Questerix Fractions to the Next Level

**Date:** 2026-05-04
**Status:** Draft master plan — coordinates the 15 dated plans below.
**Scope:** Smooth UX foundations • Touch & Mobile • Pedagogy • Content • Reliability • Validation.

This is the umbrella plan. It does **not** duplicate per-plan detail; each row below points to the authoritative document. Update this file when phase status changes; never branch off two roadmaps.

**What this file is:** the single sequencing source of truth across all dated plans. It tracks plan status, phase order, cross-plan blockers, and the gates that close each phase. Per-plan detail (problem statement, phases, acceptance criteria, files touched) lives in the per-plan document — this file links to them. When sequencing decisions change (e.g., plan X must wait for plan Y), update this file first; per-plan dependencies tables refer back here.

**Plan status states:** every per-plan document declares one of: `Draft` (problem and phases written, no work started), `Ready` (sign-off received, branch named, ready to start), `In Progress (branch: <name>)`, `Merged`, or `Abandoned (reason: <why>)`. Update the registry below in lockstep.

---

## Why now

The MVP gates are green (persistence, resumption, backup verified 2026-05-01). What's still in the way of a "next level" experience:

1. **No shared interaction grammar.** Each scene picks its own timings, easings, and gesture thresholds; UI quality drifts every PR.
2. **Mobile UX friction.** Inconsistent hit regions, untested 360 px layouts, drag responsiveness never measured on real touch hardware.
3. **Pedagogy ceiling.** Misconception detectors underused, hint system shallow, no escape hatch for stuck students, mastery invisible to learners, mastered ≠ remembered.
4. **Content thinness.** Narrow archetype distribution per level, limited question variety inside each level.
5. **Reliability cliffs.** Single Dexie corruption bricks the app; Phaser scene throws give a blank canvas; no real-device or cross-browser test surface; the bundle will exceed 1.0 MB the moment plans 5 and 7 ship.
6. **No validation loop.** This is a validation project (C10), but no protocol describes how Ryan turns the work into a verdict on whether the app teaches K–2 children fractions.

Each is addressed by one or more of the plans below, sequenced so file conflicts stay minimal and the foundational work lands first.

## Phase 0 — Foundation (lands before anything else touches UI)

**Plan 0 (interaction-and-motion-system)** ships first. It is the contract every later plan inherits from:

- `motion.ts` durations, easings, distances → all later tweens go through it.
- `interaction.ts` gesture grammar → every drag/tap/long-press follows the same thresholds.
- `states.ts` visual state language → every button looks the same in idle/hover/pressed/focused/disabled/error.
- `feedbackBus.ts` → every "correct / incorrect / snap" emission fires visual + audio + motion in lockstep.

Without this, the other 15 plans each invent their own answers and the app feels stitched together. Acceptance gate: 0 unguarded `tweens.add` calls in the source tree, gesture tokens consumed by all 10 archetypes, state language baseline screenshot at 360 / 768 / 1024 committed.

## Phase 1 — Audit & baselines (read-only, parallel)

Every workstream has its own audit phase. Run them **first**, in parallel, before any source change beyond plan 0:

- Plan 1 Phase 1 — button inventory.
- Plan 2 Phase 1–2 — touch target + font + reduced-motion checklist and component audit.
- Plan 3 Phase 0 — orphan-asset audit.
- Plan 4 Phase 1–2 — perf instrumentation + baseline measurements.
- Plan 5 Phase 1 — Playwright WebKit project added (CI surface ready before remediation).
- Plan 7 Phase 1 — misconception detector coverage report.
- Plan 13 Phase 1 — archetype distribution matrix and question-variety gap list.
- Plan 15 Phases 1–4 — pilot protocol documents drafted (no children yet).

Output: a single `audit/` directory holding inventory JSONs, screenshot baselines, perf numbers, gap reports, and pilot doc drafts. These are the inputs to Phase 2 below.

## Phase 2 — Touch, perf, and reliability hardening

Sequenced because plans 1, 2, 3, 4 all touch the same scene files (MenuScene, LevelMapScene, SettingsScene, OnboardingScene, all 10 interactions).

1. **Plan 1 (button-hit-regions)** lands first — establishes the padded-rectangle mechanism and the shared helper at the 3+ duplicate threshold (its Phase 5).
2. **Plan 2 (touchscreen-a11y) Phase 3 onward** waits for plan 1's Phase 4 to merge to main (its hard gate). Phases 1–2, 5 of plan 2 may run in parallel as audit-only.
3. **Plan 4 (performance-and-drag-latency) Phase 3** runs in parallel with plan 2 — different files, same goal of "drags feel right".
4. **Plan 3 (visual-audit) Phase 0 (asset cleanup)** runs anytime. Phases 1–2 (capture baselines) run **after** plans 1, 2, 4 land so baselines reflect the corrected UI.
5. **Plan 5 (cross-browser matrix)** lights up WebKit in CI early in this phase so plans 1–4 are validated on Safari from their first PR.
6. **Plan 6 (crash-and-recovery)** runs in parallel with everything in this phase — independent file footprint.

Acceptance gate to leave Phase 2: all interactive elements ≥ 44×44 CSS px, body text ≥ 16 px, drag targets ≥ 48 px on the smaller axis, multi-touch enabled, no horizontal overflow at 360 px, P95 drag latency ≤ 50 ms on budget profile, P95 frame ≤ 20 ms steady-state, recovery scenes catch every uncaught throw, Playwright WebKit + Chromium green, visual baselines committed at 360 px.

## Phase 3 — Pedagogy depth

These plans touch overlapping files (`HintLadder`, `Level01Scene`, `LevelScene`, BKT, persistence). Sequence them:

1. **Plan 7 (misconception-and-hint-system)** lands first — expands detectors and enriches hint copy, which becomes the input data plan 8 and plan 10 need.
2. **Plan 8 (worked-example-flow)** lands after plan 7 because the demo CTA references the same `hintLadder.state.exhausted` signal that plan 7 hardens, and the demo CTA must coexist with the deepened hint copy without overlapping at 360 px.
3. **Plan 9 (progression-mastery-display)** can run in parallel with plan 8 because its file footprint (`ProgressBar`, `SessionCompleteOverlay`, `skillMastery` repo) does not overlap with plan 8's surface.
4. **Plan 10 (spaced-repetition)** lands after plan 9 — depends on its mastery state model.
5. **Plan 11 (screen-reader-keyboard-parity)** runs in parallel; its surface is the A11yLayer mount order across all scenes affected by plans 7–9.
6. **Plan 12 (multi-student-and-first-run)** runs in parallel; independent file footprint.

Acceptance gate to leave Phase 3: ≥ 15 distinct misconceptions trigger across an L1–L9 synthetic playthrough, "Show me how" appears on the correct gate and requires re-attempt, BKT cannot reach `MASTERED` from demo-assisted answers, end-of-level mastery summary renders, warm-up loop functional, every scene's announcement order documented, multi-student switching works, every new tween has a `prefers-reduced-motion` guard.

## Phase 4 — Content, performance polish, and integration

1. **Plan 13 (content-archetype-expansion)** runs largely independent of Phases 0–3 because it edits `LEVEL_META`, runs the pipeline, and touches the curriculum bundle — none of which conflict with scene/component code. Starts as soon as plan 7 ships its expanded hint catalog.
2. **Plan 14 (bundle-splitting-and-cold-start)** lands after plans 7 and 13 land because the bundle size we are splitting is only realistic after their content + hint expansion. Coordinate the curriculum split (plan 14 Phase 2) so the pipeline regeneration in plan 13 lands on the new layout.
3. Run full suite: `typecheck`, `lint`, `test:unit`, `test:e2e` (Chromium + WebKit), `test:a11y`, `validate:curriculum`, `measure-bundle`, `npm run perf`.
4. Re-capture visual baselines (plan 3 Phase 2) on top of all merged changes; commit baseline updates in the integration PR.
5. Phase-close: append cross-cutting learnings to `.claude/learnings.md`, update `docs/00-foundation/decision-log.md` with any architectural decisions made (BKT assistance gating, hit-area helper API, gesture grammar, splitting boundaries), run `npm run sync:claude-md`.

## Phase 5 — Validation loop (continuous, not a phase end)

**Plan 15 (pilot-protocol)** is the project's reason for existing under C10. It does not end:

- **Round 1** — after Phase 2 closes: 3 children, friction-driven; results triaged into plans 7, 9, 13.
- **Round 2** — after Phase 3 closes: 5 children, comprehension-driven; pre/post deltas drive plan 13 content tuning.
- **Round 3** — after Phase 4 closes: 5 children, retention-driven; results validate or reset plan 10.

Every plan's PR description must answer: "Does this respond to a pilot finding, or speculate?" If speculate, it lives behind a feature flag until a pilot validates it (extends the env-gated observability discipline already in place).

## Constraints

All 15 plans operate inside the locked C1–C10 envelope. Notable touchpoints:

- **C1** — plans 5, 6, 14, 15 all stay client-side; no telemetry escapes the device unless `VITE_OTLP_URL` is set.
- **C5** — no plan introduces new localStorage keys; plan 9 mastery summaries persist via Dexie, plan 12 student profiles via Dexie.
- **C7** — plan 5 makes the iOS Safari + Android Chrome support real, not aspirational.
- **C8** — plan 13 enforces halves → thirds → fourths progression.
- **C9** — plan 8's worked-example demos cap at 2 s; plan 10's warm-up caps at 3 questions so 15-min/level survives.
- **C10** — plan 3 reframes screenshots as Playwright `toHaveScreenshot()` regression baselines, plan 15 makes the validation loop explicit.
